# output-capture Specification

## Purpose
Make the output of remote commands legible to an LLM without blowing its context window. Every `session_run` invocation pipes its combined stdout+stderr to a per-command log file on the remote (`~/.tai-ssh/logs/<log_id>.log`) and returns a size-bounded head/tail slice; the full content stays on the remote so the LLM can grep, tail, or cat it later via the same `session_run` surface. No dedicated log-reading tool exists — standard shell utilities against `log_path` are the contract.
## Requirements
### Requirement: On-remote log file per command
Every command run via `session_run` SHALL have its combined stdout and stderr captured to a file on the remote host at `~/.tai-ssh/logs/<log_id>.log`, where `<log_id>` is a ULID generated by the MCP before the command is sent.

#### Scenario: Log directory created on first use
- **WHEN** `~/.tai-ssh/logs/` does not exist on the remote at command time
- **THEN** the MCP SHALL create the directory (with mode `0700`) before running the command

#### Scenario: log_id returned to caller
- **WHEN** a `session_run` returns any status (`done`, `still_running`, `needs_*`)
- **THEN** the response SHALL include `log_id` and `log_path`
- **AND** `log_path` SHALL be an absolute remote path

### Requirement: Head/tail slicing with size threshold
Tool responses SHALL include `head` and `tail` slices of the captured output. If the total captured output is at or below the threshold (default 4 KB and 50 lines), the full output SHALL be inlined with `truncated: false`. Otherwise `truncated` SHALL be `true` and each of `head`/`tail` SHALL contain at most the threshold-sized slice.

#### Scenario: Small output returned in full
- **WHEN** a command produces 12 lines totalling 800 bytes
- **THEN** the response SHALL have `truncated: false` and `head` containing the full output, with `tail` either empty or equal to `head`

#### Scenario: Large output returned as head + tail
- **WHEN** a command produces 100 KB of output
- **THEN** the response SHALL have `truncated: true`
- **AND** `head` SHALL contain at most ~2 KB of the first lines
- **AND** `tail` SHALL contain at most ~2 KB of the last lines
- **AND** `bytes` SHALL equal the total byte count of captured output

### Requirement: LLM reads full logs via existing shell tools
The MCP SHALL NOT provide a dedicated log-reading tool. The LLM SHALL read additional portions of a previous command's output by invoking `session_run` against the same host with standard utilities (`tail`, `head`, `grep`, `cat`, `awk`, `wc`) against `log_path`.

#### Scenario: LLM tails a previous log
- **WHEN** the LLM calls `session_run("pi-living/default", "tail -200 ~/.tai-ssh/logs/01JV3N....log")`
- **THEN** the call SHALL behave like any other `session_run`, including its own audit record and its own (smaller) `log_id`

### Requirement: Log retention sweep
The MCP SHALL delete log files older than the configured retention period (default 7 days, per-host overridable via `log_retention_days` in `hosts.toml`) from `~/.tai-ssh/logs/` on every initial connect to a host. The sweep SHALL be best-effort and SHALL NOT block command execution.

#### Scenario: Old logs removed on connect
- **WHEN** the MCP connects to a host with files older than the retention period in `~/.tai-ssh/logs/`
- **THEN** those files SHALL be deleted
- **AND** the count of deleted files SHALL be recorded in the audit log as a `housekeeping` event

#### Scenario: Sweep failure does not abort command
- **WHEN** the retention sweep fails (e.g., permission denied)
- **THEN** the failure SHALL be logged to the audit log
- **AND** the connection and subsequent command SHALL proceed normally
